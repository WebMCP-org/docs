`@mcp-b/webmcp-types` provides TypeScript type definitions for the entire WebMCP surface: the [W3C standard API](https://webmachinelearning.github.io/webmcp/), the MCP-B extension layer, and a JSON Schema inference engine that derives argument types from schema literals at compile time. It ships zero runtime code and zero side effects.

```bash
npm install --save-dev @mcp-b/webmcp-types
```

## Activation

TypeScript may not automatically include global declarations from npm packages. Use one of these methods:

<Tabs>
  <Tab title="tsconfig.json">
    ```json
    {
      "compilerOptions": {
        "types": ["@mcp-b/webmcp-types"]
      }
    }
    ```
  </Tab>
  <Tab title="Triple-slash reference">
    ```ts
    /// <reference types="@mcp-b/webmcp-types" />
    ```
  </Tab>
  <Tab title="Type-only import">
    ```ts
    import type {} from '@mcp-b/webmcp-types';
    ```
  </Tab>
</Tabs>

After activation, `navigator.modelContext` is typed as `ModelContext` and `navigator.modelContextTesting` as `ModelContextTesting | undefined`.

## The type layering

The package mirrors the [runtime layering](/explanation/architecture/runtime-layering) of WebMCP itself. Four interfaces form the core hierarchy:

| Interface | Layer | What it covers |
|-----------|-------|----------------|
| `ModelContextCore` | W3C standard | `registerTool()`, `unregisterTool()`, `provideContext()`, `clearContext()` |
| `ModelContextExtensions` | MCP-B extension | `listTools()`, `callTool()`, `addEventListener()` for `toolcall` and `toolschanged` events |
| `ModelContext` | Alias for `ModelContextCore` | The type of `navigator.modelContext` |
| `ModelContextWithExtensions` | `ModelContextCore & ModelContextExtensions` | The full runtime shape when `@mcp-b/global` is loaded |

`ModelContextCore` types the four methods defined in the [W3C WebMCP specification](https://webmachinelearning.github.io/webmcp/). Everything on `ModelContextExtensions` is an MCP-B addition (not part of the standard). For a deeper discussion, see [Strict core vs MCP-B extensions](/explanation/strict-core-vs-mcp-b-extensions).

```ts
import type {
  ModelContext,
  ModelContextWithExtensions,
} from '@mcp-b/webmcp-types';

// Strict core only (what the W3C spec defines)
const core: ModelContext = navigator.modelContext;
core.registerTool({ name: 'ping', description: 'Health check', execute: () => ({ content: [] }) });

// Full runtime (after @mcp-b/global initializes)
const full = navigator.modelContext as unknown as ModelContextWithExtensions;
const tools = full.listTools();
```

## Tool descriptors

`ToolDescriptor` is the main interface for defining tools. It accepts three generic parameters for type-safe argument inference, result typing, and name narrowing:

```ts
import type { ToolDescriptor, CallToolResult } from '@mcp-b/webmcp-types';

type SearchTool = ToolDescriptor<
  { query: string; limit?: number },  // TArgs
  CallToolResult,                      // TResult
  'search'                             // TName
>;
```

`ToolListItem` has the same shape as `ToolDescriptor` but without the `execute` function. It represents tool metadata returned by `listTools()`.

## JSON Schema inference

The package infers TypeScript types from JSON Schema literals at compile time. Write a schema with `as const satisfies JsonSchemaForInference`, and `registerTool` infers the `execute` callback's argument type automatically:

```ts
import type { JsonSchemaForInference } from '@mcp-b/webmcp-types';

const inputSchema = {
  type: 'object',
  properties: {
    query: { type: 'string' },
    limit: { type: 'integer', minimum: 1, maximum: 50 },
  },
  required: ['query'],
  additionalProperties: false,
} as const satisfies JsonSchemaForInference;

navigator.modelContext.registerTool({
  name: 'search',
  description: 'Search indexed docs',
  inputSchema,
  async execute(args) {
    // args is inferred as { query: string; limit?: number }
    return {
      content: [{ type: 'text', text: `Searching for ${args.query}` }],
    };
  },
});
```

You can also extract the inferred type directly with `InferArgsFromInputSchema`:

```ts
import type { InferArgsFromInputSchema } from '@mcp-b/webmcp-types';

type Args = InferArgsFromInputSchema<typeof inputSchema>;
// { query: string; limit?: number }
```

The inference engine supports `type`, `properties`, `required`, `items`, `enum`, `const`, `nullable`, and `additionalProperties`. Other JSON Schema keywords are accepted as metadata but do not affect the inferred type. If a schema is widened at runtime (for example, typed as `InputSchema` instead of a literal), inference falls back to `Record<string, unknown>`.

For practical patterns and examples, see [Use schemas and structured output](/how-to/use-schemas-and-structured-output).

## Typed model context

When your project has a fixed set of known tools, `TypedModelContext` narrows `callTool` and `listTools` to those specific tool names and argument types:

```ts
import type {
  CallToolResult,
  ToolDescriptor,
  TypedModelContext,
} from '@mcp-b/webmcp-types';

type SearchTool = ToolDescriptor<
  { query: string; limit?: number },
  CallToolResult & { structuredContent: { total: number } },
  'search'
>;
type PingTool = ToolDescriptor<Record<string, never>, CallToolResult, 'ping'>;

type AppContext = TypedModelContext<readonly [SearchTool, PingTool]>;

declare const ctx: AppContext;

// Name is narrowed; arguments are type-checked
await ctx.callTool({ name: 'search', arguments: { query: 'webmcp' } });

// Tools with no args make `arguments` optional
await ctx.callTool({ name: 'ping' });
```

## When to use this package directly

Most users get these types transitively through [`@mcp-b/global`](/packages/global/overview) or [`@mcp-b/webmcp-polyfill`](/packages/webmcp-polyfill/overview). Install `@mcp-b/webmcp-types` as a direct dev dependency when:

- You are building a library that types against the WebMCP surface without choosing a runtime.
- You need compile-time inference from JSON Schema literals.
- You want `navigator.modelContext` types in TypeScript without any runtime code.

If you need a runtime implementation, use [`@mcp-b/webmcp-polyfill`](/packages/webmcp-polyfill/overview) (strict core only) or [`@mcp-b/global`](/packages/global/overview) (full stack with MCP-B extensions).

## API reference

### Navigator augmentation

| Property | Type | Description |
|----------|------|-------------|
| `navigator.modelContext` | `ModelContext` | Strict core WebMCP surface |
| `navigator.modelContextTesting` | `ModelContextTesting \| undefined` | Testing API (optional) |

### Core interfaces

#### `ModelContext`

Alias for `ModelContextCore`. This is the type of `navigator.modelContext`.

#### `ModelContextCore`

The strict specification surface.

| Method | Signature |
|--------|-----------|
| `registerTool` | Multiple overloads (see below) |
| `unregisterTool` | `(name: string) => void` |

#### `ModelContextExtensions`

MCP-B extension methods. Not part of the WebMCP specification.

| Method | Signature |
|--------|-----------|
| `listTools` | `() => ToolListItem[]` |
| `callTool` | `(params: { name: string; arguments?: Record<string, unknown> }) => Promise<ToolResponse>` |
| `addEventListener` | `(type: 'toolcall' \| 'toolschanged', listener, options?) => void` |
| `removeEventListener` | `(type: 'toolcall' \| 'toolschanged', listener, options?) => void` |
| `dispatchEvent` | `(event: Event) => boolean` |

#### `ModelContextWithExtensions`

Combines `ModelContextCore & ModelContextExtensions`. Use when you need both strict core and extension methods.

### Tool types

#### `ToolDescriptor<TArgs, TResult, TName>`

| Property | Type | Required |
|----------|------|----------|
| `name` | `TName` | Yes |
| `description` | `string` | Yes |
| `inputSchema` | `InputSchema` | No |
| `outputSchema` | `InputSchema` | No |
| `annotations` | `ToolAnnotations` | No |
| `execute` | `(args: TArgs, client: ModelContextClient) => MaybePromise<ToolExecuteResult<TResult>>` | Yes |

#### `ToolListItem<TName>`

Tool metadata returned by `listTools()`. Same shape as `ToolDescriptor` without `execute`.

#### `ToolAnnotations`

| Property | Type | Description |
|----------|------|-------------|
| `title` | `string?` | Display title |
| `readOnlyHint` | `boolean \| 'true' \| 'false'` | Read-only indicator |
| `destructiveHint` | `boolean \| 'true' \| 'false'` | Destructive action indicator |
| `idempotentHint` | `boolean \| 'true' \| 'false'` | Idempotent action indicator |
| `openWorldHint` | `boolean \| 'true' \| 'false'` | External system access indicator |

#### `CallToolResult` / `ToolResponse`

```ts
interface CallToolResult {
  content: Array<ContentBlock | LooseContentBlock>;
  structuredContent?: JsonObject;
  isError?: boolean;
}

type ToolResponse = CallToolResult;
```

### Schema inference rules

Supported JSON Schema keywords for inference:

| Keyword | Effect |
|---------|--------|
| `type` | Determines base TypeScript type |
| `properties` | Creates named fields |
| `required` | Marks fields as non-optional |
| `items` | Array element type |
| `enum` | Union of literal values |
| `const` | Single literal value |
| `nullable` | Adds `null` to the type |
| `additionalProperties` | Controls extra key acceptance |

#### `additionalProperties` behavior

| Schema shape | Inferred extras |
|-------------|----------------|
| `additionalProperties: false` | No extra keys |
| `additionalProperties` omitted or `true` | Extra keys as `unknown` |
| `additionalProperties: { ... }` with no named `properties` | `Record<string, ...>` |
| `additionalProperties: { ... }` with named `properties` | Named properties inferred, extras as `unknown` |

### Output schema inference

#### `ToolResultFromOutputSchema<T>`

When `outputSchema` is a literal object schema, `structuredContent` is narrowed.

```ts
import type { JsonSchemaForInference } from '@mcp-b/webmcp-types';

const outputSchema = {
  type: 'object',
  properties: {
    total: { type: 'integer' },
    items: { type: 'array', items: { type: 'string' } },
  },
  required: ['total'],
  additionalProperties: false,
} as const satisfies JsonSchemaForInference;

navigator.modelContext.registerTool({
  name: 'search',
  description: 'Search docs',
  inputSchema,
  outputSchema,
  async execute(args) {
    return {
      content: [{ type: 'text', text: 'done' }],
      structuredContent: {
        total: 1,            // required, inferred as number
        items: ['result'],   // optional, inferred as string[]
      },
    };
  },
});
```

#### `ToolDescriptorFromSchema<TInput, TOutput, TName>`

Schema-driven descriptor type. Both `execute` args and `structuredContent` are inferred from the schemas.

### Typed model context helpers

| Type | Description |
|------|-------------|
| `InferToolArgs<T>` | Extract args type from a tool descriptor |
| `InferToolResult<T>` | Extract result type from a tool descriptor |
| `ToolName<TTools>` | Union of tool names from a descriptor array |
| `ToolByName<TTools, TName>` | Extract a descriptor by name |
| `ToolArgsByName<TTools, TName>` | Args type by tool name |
| `ToolResultByName<TTools, TName>` | Result type by tool name |
| `ToolCallParams<TName, TArgs>` | Typed parameters for `callTool` |

### Testing types

#### `ModelContextTesting`

| Method | Signature |
|--------|-----------|
| `listTools` | `() => ModelContextTestingToolInfo[]` |
| `executeTool` | Overloaded: JSON string or streamed source |
| `registerToolsChangedCallback` | `(callback: () => void) => void` |
| `getCrossDocumentScriptToolResult` | `() => Promise<string>` |

#### `ModelContextTestingToolInfo`

| Property | Type |
|----------|------|
| `name` | `string` |
| `description` | `string` |
| `inputSchema` | `string?` (JSON-serialized) |

### Other exports

| Export | Description |
|--------|-------------|
| `InputSchema` | JSON Schema interface for tool parameters |
| `ContentBlock` | Discriminated union of content types |
| `LooseContentBlock` | Permissive content block type |
| `MaybePromise<T>` | `T \| Promise<T>` |
| `ModelContextClient` | Execute callback client with `requestUserInteraction` |
| `ToolExecutionContext` | Extended client with `elicitInput` |
| `ModelContextInput` | Legacy options type (kept for backwards compatibility) |
| `ModelContextOptions` | Alias for `ModelContextInput` |
| `ElicitationParams` | Elicitation request parameters |
| `ElicitationResult` | Elicitation response |
| `ToolCallEvent` | Event dispatched on tool invocation |
