`@mcp-b/smart-dom-reader` turns a live DOM into structured data that AI agents can act on. It extracts interactive elements (buttons, links, inputs, forms), generates ranked CSS selectors, detects page landmarks, and keeps token usage low by omitting irrelevant content.

The package has zero runtime dependencies and works in any browser context: page scripts, content scripts, or injected bundles.

## How it works

The extraction pipeline runs in three stages:

1. **Detect landmarks.** `ContentDetection` identifies the page's header, navigation, main content area, sidebars, and footer using semantic HTML, ARIA roles, and a scoring algorithm that weighs paragraph density, heading presence, and link density.

2. **Traverse and extract elements.** `DOMTraversal` walks the DOM within the detected container, pulling out interactive elements (buttons, links, inputs, forms) and, in `full` mode, semantic elements (headings, images, tables, lists, articles). It checks visibility, viewport bounds, and any custom filters before including each element.

3. **Generate ranked selectors.** `SelectorGenerator` produces multiple selector candidates for every extracted element and scores them by stability:

| Strategy | Score | Example |
|----------|-------|---------|
| ID | 100 | `#unique-id` |
| `data-testid` | 90 | `[data-testid="submit"]` |
| ARIA (role + label) | 85 | `[role="button"][aria-label="Submit"]` |
| Name attribute | 78 | `input[name="email"]` |
| Class path | 50-78 | `.form-container .submit-btn` |
| XPath | 40 | `//form/button[1]` |
| Text-based | 30 | `button:contains("Sign in")` |

The highest-scoring candidate becomes the primary `css` selector. All candidates are included in the output so agents can fall back to alternatives when the page structure changes.

## Quick start

```bash
npm install @mcp-b/smart-dom-reader
```

### One-shot extraction

Use `SmartDOMReader` when you need all elements in a single pass:

```typescript
import { SmartDOMReader } from '@mcp-b/smart-dom-reader';

// Interactive mode: buttons, links, inputs, forms
const interactive = SmartDOMReader.extractInteractive(document);

// Full mode: interactive elements + headings, images, tables, lists
const full = SmartDOMReader.extractFull(document);

// From a specific element
const region = SmartDOMReader.extractFromElement(element, 'interactive');
```

For more control, create an instance with options:

```typescript
const reader = new SmartDOMReader({
  mode: 'interactive',
  mainContentOnly: true,
  viewportOnly: false,
});
const result = reader.extract(document);
```

### Progressive extraction

Use `ProgressiveExtractor` when token budget matters. It breaks extraction into steps so an agent can decide what to drill into:

```typescript
import { ProgressiveExtractor } from '@mcp-b/smart-dom-reader';

// Step 1: Get the page outline (minimal tokens)
const structure = ProgressiveExtractor.extractStructure(document);

// Step 2: Drill into a specific region
const region = ProgressiveExtractor.extractRegion(
  structure.summary.mainContentSelector,
  document,
  { mode: 'interactive' }
);

// Step 3: Extract readable content from a region
const content = ProgressiveExtractor.extractContent(
  'article.main-article',
  document,
  { includeHeadings: true, includeLists: true }
);
```

## When to use each approach

| Approach | Best for | Token cost |
|----------|----------|------------|
| `SmartDOMReader` (one-shot) | Automation scripts, testing, small pages | Higher (full dump) |
| `ProgressiveExtractor` (staged) | AI agent workflows, large pages, conversational exploration | Lower (pay-as-you-go) |

One-shot extraction returns everything at once. Progressive extraction lets the agent first see the page structure, then request details for specific regions, which avoids wasting tokens on irrelevant parts of the page.

## Output format

### SmartDOMResult

Both approaches return a `SmartDOMResult`:

```typescript
interface SmartDOMResult {
  mode: 'interactive' | 'full';
  timestamp: number;
  page: {
    url: string;
    title: string;
    hasErrors: boolean;
    isLoading: boolean;
    hasModals: boolean;
    hasFocus?: string;         // selector of focused element
  };
  landmarks: {
    navigation: string[];      // selectors for nav regions
    main: string[];
    forms: string[];
    headers: string[];
    footers: string[];
    articles: string[];
    sections: string[];
  };
  interactive: {
    buttons: ExtractedElement[];
    links: ExtractedElement[];
    inputs: ExtractedElement[];
    forms: FormInfo[];
    clickable: ExtractedElement[];
  };
  semantic?: { /* headings, images, tables, lists, articles */ };  // full mode only
  metadata?: { totalElements, extractedElements, mainContent?, language? };  // full mode only
}
```

### ExtractedElement

Each element carries its tag, visible text, ranked selectors, relevant attributes, DOM context (nearest form, section, main, nav), and interaction flags:

```typescript
interface ExtractedElement {
  tag: string;
  text: string;
  selector: {
    css: string;
    xpath: string;
    candidates?: Array<{ type: string; value: string; score: number }>;
  };
  attributes: Record<string, string>;
  context: {
    nearestForm?: string;
    parentChain: string[];
  };
  interaction: {
    click?: boolean;
    change?: boolean;
    submit?: boolean;
    disabled?: boolean;
  };
}
```

Interaction flags use a compact format: only truthy values are included, which saves tokens.

### StructuralOverview (progressive extraction)

`extractStructure` returns a high-level map of the page:

```typescript
interface StructuralOverview {
  regions: {
    header?: RegionInfo;
    navigation?: RegionInfo[];
    main?: RegionInfo;
    sidebar?: RegionInfo[];
    footer?: RegionInfo;
    modals?: RegionInfo[];
    sections?: RegionInfo[];
  };
  forms: Array<{ selector: string; location: string; inputCount: number; purpose?: string }>;
  summary: {
    totalInteractive: number;
    totalForms: number;
    hasModals: boolean;
    hasErrors: boolean;
    isLoading: boolean;
    mainContentSelector?: string;
  };
  suggestions?: string[];    // AI-friendly hints ("Page has error indicators", etc.)
}
```

## Use cases

**Agent-driven browser automation.** An agent calls `extractStructure` to understand the page layout, then `extractRegion` to get selectors for the area it needs to interact with. The ranked selectors let the agent pick the most stable target for clicking or typing.

**Page understanding.** Extract the full page in `full` mode to get headings, images, tables, and metadata alongside interactive elements. Feed this to an LLM for summarization or question answering.

**Form filling.** `extractRegion` scoped to a form element returns all inputs with their names, types, placeholders, and validation attributes. The agent maps these to its data and fills each field using the generated selectors.

## Bundle string for injection

A self-contained IIFE bundle is available at `@mcp-b/smart-dom-reader/bundle-string` for injection into pages (for example, via `chrome.scripting.executeScript`):

```typescript
import { SMART_DOM_READER_BUNDLE } from '@mcp-b/smart-dom-reader/bundle-string';

const code = `(() => {
  ${SMART_DOM_READER_BUNDLE}
  return SmartDOMReaderBundle.executeExtraction('extractStructure', {
    selector: undefined,
    formatOptions: { detail: 'summary' }
  });
})()`;
```
