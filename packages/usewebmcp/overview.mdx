`usewebmcp` is a standalone React hook that registers and unregisters [WebMCP](https://webmachinelearning.github.io/webmcp/) tools on `navigator.modelContext`, tied to your component's lifecycle. It covers the strict-core API surface only: `registerTool` and `unregisterTool`. No prompts, resources, sampling, or other MCP-B extensions.

## How it works

When your component mounts, `useWebMCP` calls `navigator.modelContext.registerTool()` with the tool descriptor you provide. When the component unmounts, it calls `unregisterTool()` to clean up. The hook also tracks execution state (loading, results, errors) so your component can react to tool invocations from an AI assistant.

Callbacks like `execute`, `onSuccess`, and `onError` are stored in refs, so updating them does not trigger re-registration. Schema objects (`inputSchema`, `outputSchema`, `annotations`) are compared by reference. If their reference changes, the tool re-registers.

<Note>
`navigator.modelContext` must exist at runtime. Provide it through native browser support, [`@mcp-b/webmcp-polyfill`](/packages/webmcp-polyfill/overview), or [`@mcp-b/global`](/packages/global/overview). See [Choosing a runtime](/how-to/choose-runtime) for guidance.
</Note>

## Quick start

Install the package alongside React:

```bash
pnpm add usewebmcp react
```

Register a tool inside a React component:

```tsx
import { useWebMCP } from 'usewebmcp';

const OUTPUT_SCHEMA = {
  type: 'object',
  properties: {
    count: { type: 'number' },
  },
} as const;

export function CounterTool() {
  const { state } = useWebMCP({
    name: 'counter_get',
    description: 'Get the current count',
    outputSchema: OUTPUT_SCHEMA,
    execute: async () => ({ count: 42 }),
  });

  return <div>Last result: {state.lastResult?.count ?? 'none'}</div>;
}
```

The hook returns `{ state, execute, reset }`. See the [API reference](#api-reference) below for the full config fields, state shape, and type inference details.

## What it supports

`usewebmcp` covers the strict-core WebMCP surface:

- **`registerTool`** on mount, with your tool descriptor
- **`unregisterTool`** on unmount or when registration dependencies change
- **Execution state tracking** (`isExecuting`, `lastResult`, `error`, `executionCount`)
- **Input type inference** from JSON Schema (`as const`) or Standard Schema v1 (Zod v4, Valibot, ArkType)
- **Output type inference** and `structuredContent` via `outputSchema`
- **Lifecycle callbacks** (`onSuccess`, `onError`)

It does not support `registerPrompt`, `registerResource`, `createMessage`, `elicitInput`, or any other [MCP-B extension](/explanation/strict-core-vs-mcp-b-extensions).

## When to use usewebmcp vs @mcp-b/react-webmcp

| | `usewebmcp` | `@mcp-b/react-webmcp` |
|---|---|---|
| API surface | Strict core (`registerTool` / `unregisterTool`) | Full MCP-B (tools, prompts, resources, sampling, elicitation) |
| Dependencies | `@mcp-b/webmcp-polyfill`, `@mcp-b/webmcp-types` | `@mcp-b/global` (the full runtime) |
| Provider required | No | Yes (`WebMCPProvider`) |
| Use case | Minimal tool registration, no MCP-B runtime needed | Full MCP-B integration with React |

Choose `usewebmcp` when you only need `registerTool` and want to stay on the standard-owned browser surface. Choose [`@mcp-b/react-webmcp`](/packages/react-webmcp/overview) when you need the broader MCP-B capabilities.

For more on this distinction, see [Strict core vs MCP-B extensions](/explanation/strict-core-vs-mcp-b-extensions) and the [package layering explanation](/explanation/architecture/runtime-layering).

## API reference

### `useWebMCP(config, deps?)`

```ts
import { useWebMCP } from 'usewebmcp';
```

| Parameter | Type | Description |
|---|---|---|
| `config` | `WebMCPConfig` | Tool configuration object |
| `deps` | `DependencyList` | Optional dependency array that triggers tool re-registration when values change |

**Returns:** `WebMCPReturn`

| Field | Type | Description |
|---|---|---|
| `state` | `ToolExecutionState` | Current execution state |
| `execute(input)` | `(input) => Promise` | Manually invoke the tool from your UI or tests |
| `reset()` | `() => void` | Reset state to initial values |

### Config fields

| Field | Type | Required | Description |
|---|---|---|---|
| `name` | `string` | Yes | Unique tool identifier (lowercase with underscores) |
| `description` | `string` | Yes | Human-readable description for AI assistants |
| `inputSchema` | JSON Schema or Standard Schema v1 | No | Defines input parameters. Use `as const` for inference. |
| `outputSchema` | `JsonSchemaObject` | No | Defines output structure and enables `structuredContent`. |
| `annotations` | `ToolAnnotations` | No | Metadata hints such as `readOnlyHint` and `idempotentHint` |
| `execute(input)` | `(input) => T \| Promise<T>` | Yes* | Preferred tool implementation function |
| `handler(input)` | `(input) => T \| Promise<T>` | Yes* | Backward-compatible alias for `execute` |
| `formatOutput(output)` | `(output) => string` | No | Deprecated. Prefer `outputSchema`. |
| `onSuccess(result, input)` | `(result, input) => void` | No | Callback after successful execution |
| `onError(error, input)` | `(error, input) => void` | No | Callback after failed execution |

*One of `execute` or `handler` is required. If both are present, `execute` wins.*

### `ToolExecutionState`

| Field | Type | Description |
|---|---|---|
| `isExecuting` | `boolean` | Whether the tool is currently running |
| `lastResult` | `TOutput \| null` | Result from the most recent successful execution |
| `error` | `Error \| null` | Error from the most recent failed execution |
| `executionCount` | `number` | Total successful executions |

### Type inference

Input inference works with two schema formats:

- JSON Schema literals with `as const`
- Standard Schema v1 implementations such as Zod v4, Valibot, and ArkType

Output inference comes from `outputSchema` when you pass a JSON Schema literal:

- the `execute` or `handler` return type is inferred from the schema
- `state.lastResult` is typed accordingly
- the MCP response includes `structuredContent`

```tsx
const OUTPUT_SCHEMA = {
  type: 'object',
  properties: {
    total: { type: 'integer' },
  },
  required: ['total'],
  additionalProperties: false,
} as const;

const tool = useWebMCP({
  name: 'count_items',
  description: 'Count items',
  outputSchema: OUTPUT_SCHEMA,
  execute: () => ({ total: 3 }),
});
```

### Re-registration triggers

The hook unregisters and re-registers when any of these values change:

- `name`
- `description`
- `inputSchema` reference
- `outputSchema` reference
- `annotations` reference
- values in `deps`

These callbacks are stored in refs and do not trigger re-registration:

- `execute`
- `handler`
- `onSuccess`
- `onError`
- `formatOutput`

<Warning>
Define `inputSchema`, `outputSchema`, and `annotations` outside the component or memoize them. Fresh object references on every render force unnecessary re-registration.
</Warning>

### Output schema contract

When `outputSchema` is defined, the implementation must return a JSON-serializable object. Returning a non-object value causes an error response from the registered MCP tool.

### Exported types

| Type | Description |
|---|---|
| `WebMCPConfig` | Configuration object type |
| `WebMCPReturn` | Return value type |
| `ToolExecutionState` | Execution state shape |
| `ToolExecuteFunction` | Tool implementation function signature |
| `InferToolInput` | Infer input type from schema |
| `InferOutput` | Infer output type from schema |
