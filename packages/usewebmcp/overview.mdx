`usewebmcp` is a standalone React hook that registers and unregisters [WebMCP](https://webmachinelearning.github.io/webmcp/) tools on `navigator.modelContext`, tied to your component's lifecycle. It covers the strict-core API surface only: `registerTool` and `unregisterTool`. No prompts, resources, sampling, or other MCP-B extensions.

## How it works

When your component mounts, `useWebMCP` calls `navigator.modelContext.registerTool()` with the tool descriptor you provide. When the component unmounts, it calls `unregisterTool()` to clean up. The hook also tracks execution state (loading, results, errors) so your component can react to tool invocations from an AI assistant.

Callbacks like `execute`, `onSuccess`, and `onError` are stored in refs, so updating them does not trigger re-registration. Schema objects (`inputSchema`, `outputSchema`, `annotations`) are compared by reference. If their reference changes, the tool re-registers.

<Note>
`navigator.modelContext` must exist at runtime. Provide it through native browser support, [`@mcp-b/webmcp-polyfill`](/packages/webmcp-polyfill/overview), or [`@mcp-b/global`](/packages/global/overview). See [Choosing a runtime](/how-to/choose-runtime) for guidance.
</Note>

## Quick start

Install the package alongside React:

```bash
pnpm add usewebmcp react
```

Register a tool inside a React component:

```tsx
import { useWebMCP } from 'usewebmcp';

const OUTPUT_SCHEMA = {
  type: 'object',
  properties: {
    count: { type: 'number' },
  },
} as const;

export function CounterTool() {
  const { state } = useWebMCP({
    name: 'counter_get',
    description: 'Get the current count',
    outputSchema: OUTPUT_SCHEMA,
    execute: async () => ({ count: 42 }),
  });

  return <div>Last result: {state.lastResult?.count ?? 'none'}</div>;
}
```

The hook returns `{ state, execute, reset }`. See the [reference page](/packages/usewebmcp/reference) for the full config fields, state shape, and type inference details.

## What it supports

`usewebmcp` covers the strict-core WebMCP surface:

- **`registerTool`** on mount, with your tool descriptor
- **`unregisterTool`** on unmount or when registration dependencies change
- **Execution state tracking** (`isExecuting`, `lastResult`, `error`, `executionCount`)
- **Input type inference** from JSON Schema (`as const`) or Standard Schema v1 (Zod v4, Valibot, ArkType)
- **Output type inference** and `structuredContent` via `outputSchema`
- **Lifecycle callbacks** (`onSuccess`, `onError`)

It does not support `registerPrompt`, `registerResource`, `createMessage`, `elicitInput`, or any other [MCP-B extension](/explanation/strict-core-vs-mcp-b-extensions).

## When to use usewebmcp vs @mcp-b/react-webmcp

| | `usewebmcp` | `@mcp-b/react-webmcp` |
|---|---|---|
| API surface | Strict core (`registerTool` / `unregisterTool`) | Full MCP-B (tools, prompts, resources, sampling, elicitation) |
| Dependencies | `@mcp-b/webmcp-polyfill`, `@mcp-b/webmcp-types` | `@mcp-b/global` (the full runtime) |
| Provider required | No | Yes (`WebMCPProvider`) |
| Use case | Minimal tool registration, no MCP-B runtime needed | Full MCP-B integration with React |

Choose `usewebmcp` when you only need `registerTool` and want to stay on the standard-owned browser surface. Choose [`@mcp-b/react-webmcp`](/packages/react-webmcp/overview) when you need the broader MCP-B capabilities.

For more on this distinction, see [Strict core vs MCP-B extensions](/explanation/strict-core-vs-mcp-b-extensions) and the [package layering explanation](/explanation/architecture/runtime-layering).
