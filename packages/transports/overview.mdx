`@mcp-b/transports` provides transport classes that move MCP JSON-RPC messages across browser boundaries. Each class implements the MCP `Transport` interface and handles message serialization, origin validation, and connection lifecycle for a specific boundary type: same-tab `postMessage`, parent-to-iframe `postMessage`, Chrome extension port messaging, or user script port messaging.

## Transport types

Every boundary requires a **server transport** (where the MCP server runs) and a **client transport** (where the MCP client connects from). The two sides pair up through a shared handshake protocol.

| Boundary | Server transport | Client transport | Messaging API |
|----------|-----------------|-----------------|---------------|
| Same tab | `TabServerTransport` | `TabClientTransport` | `window.postMessage` |
| Parent and iframe | `IframeChildTransport` (inside iframe) | `IframeParentTransport` (in parent) | `window.postMessage` |
| Extension internal | `ExtensionServerTransport` (background) | `ExtensionClientTransport` (sidebar/popup) | `chrome.runtime.Port` |
| Extension to extension | `ExtensionServerTransport` (host) | `ExtensionClientTransport` with `extensionId` (guest) | `chrome.runtime.Port` |
| User script to extension | `UserScriptServerTransport` (background) | `UserScriptClientTransport` (user script) | `chrome.runtime.Port` |

### Tab transports

`TabServerTransport` and `TabClientTransport` communicate within a single browser tab using `window.postMessage`. The server listens for incoming messages and broadcasts a `mcp-server-ready` signal on start. The client waits for that signal before sending any messages.

```typescript
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { TabServerTransport } from "@mcp-b/transports";

const server = new McpServer({ name: "my-app", version: "1.0.0" });
const transport = new TabServerTransport({
  allowedOrigins: ["https://example.com"],
});

await server.connect(transport);
```

The server transport tracks pending requests and sends interrupted responses during `beforeunload` if the page navigates while a tool call is in progress. Stale requests are cleaned up after 5 minutes.

The client transport includes a configurable request timeout (default 10 seconds). If the server does not respond in time, the client synthesizes a JSON-RPC error with code `-32000`.

### Iframe transports

`IframeChildTransport` runs inside the iframe (server side) and `IframeParentTransport` runs in the parent page (client side). Both use `window.postMessage` with retry-based ready handshakes to handle iframe loading timing.

```typescript "Server inside the iframe"
import { IframeChildTransport } from "@mcp-b/transports";

const transport = new IframeChildTransport({
  allowedOrigins: ["https://parent-app.com"],
});
await server.connect(transport);
```

```typescript "Client in the parent page"
import { IframeParentTransport } from "@mcp-b/transports";

const iframe = document.querySelector("iframe");
const transport = new IframeParentTransport({
  iframe,
  targetOrigin: "https://iframe-app.com",
});
await client.connect(transport);
```

The parent transport retries the ready handshake every 250 ms (configurable via `checkReadyRetryMs`) until the iframe's server responds.

### Extension transports

`ExtensionServerTransport` and `ExtensionClientTransport` use `chrome.runtime.Port` for communication between extension components. The server transport runs in the background service worker. The client transport runs in content scripts, sidebars, or popups.

```typescript "Background service worker"
import { ExtensionServerTransport } from "@mcp-b/transports";

chrome.runtime.onConnect.addListener((port) => {
  if (port.name === "mcp") {
    const transport = new ExtensionServerTransport(port);
    server.connect(transport);
  }
});
```

```typescript "Sidebar or popup"
import { ExtensionClientTransport } from "@mcp-b/transports";

const transport = new ExtensionClientTransport({
  portName: "mcp",
});
await client.connect(transport);
```

The server transport sends periodic keep-alive pings (every 1 second by default) to prevent Chrome from shutting down the service worker. The client transport reconnects automatically with exponential backoff when the connection drops.

To connect across extensions, pass `extensionId` to `ExtensionClientTransport` and list the guest extension in the host's `externally_connectable` manifest entry.

### User script transports

`UserScriptServerTransport` and `UserScriptClientTransport` mirror the extension transports but use `chrome.runtime.onUserScriptConnect` on the server side. The API is identical to the extension transports, including keep-alive and auto-reconnect behavior.

## Security boundaries

All `postMessage`-based transports enforce origin checks:

- **Server transports** (`TabServerTransport`, `IframeChildTransport`) require an `allowedOrigins` array. Messages from origins not in the list are silently dropped.
- **Client transports** (`TabClientTransport`, `IframeParentTransport`) accept a `targetOrigin` string. Messages from other origins are ignored.

<Warning>
Setting `allowedOrigins` to `["*"]` or `targetOrigin` to `"*"` disables origin validation. Use specific origins in production.
</Warning>

Extension and user script transports rely on Chrome's built-in port security. Only code within the extension (or extensions listed in `externally_connectable`) can open a port.

## When to use this package directly

[`@mcp-b/global`](/packages/global/overview) auto-wires a `TabServerTransport` during initialization, so most applications never import `@mcp-b/transports` directly. Use this package when you need:

- A transport other than the default tab transport (iframe, extension, user script)
- Custom `allowedOrigins`, `channelId`, or timeout settings
- Multiple transports on the same page with different channel IDs
- Direct control over the transport lifecycle

For iframe-based tool bridging with a higher-level API, see [`@mcp-b/mcp-iframe`](/packages/mcp-iframe/overview), which wraps the iframe transports in a custom element.
