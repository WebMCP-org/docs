`@mcp-b/extension-tools` provides MCP tool classes for Chrome Extension APIs. Each class wraps a Chrome API namespace (tabs, bookmarks, history, storage, scripting, and others) and registers it as a set of MCP tools that AI agents can call. The package includes 70+ tool classes covering the full breadth of the Chrome Extensions platform.

## How it works

Every tool class extends `BaseApiTools`, which provides a consistent lifecycle:

1. **Construct** with an `McpServer` instance and an options object that gates individual actions.
2. **Call `.register()`**, which checks API availability at runtime and registers the tools on the server.
3. **Agents call tools** through the MCP protocol. Each tool validates input with Zod schemas, calls the underlying `chrome.*` API, and returns structured results.

If a Chrome API is unavailable (missing permission or wrong context), the class logs a warning and skips registration instead of throwing.

```typescript "background.ts"
import { TabsApiTools, HistoryApiTools } from '@mcp-b/extension-tools';
import { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js';

const server = new McpServer({ name: 'my-extension', version: '1.0.0' });

// Register only the tools you need
const tabs = new TabsApiTools(server, {
  listActiveTabs: true,
  createTab: true,
  closeTabs: true,
});
tabs.register();

const history = new HistoryApiTools(server);
history.register();
```

The options object controls which actions are exposed. Pass `true` to enable specific actions, or omit options to enable all actions for that API. Pass `false` to disable an action.

## DOM extraction and smart-dom-reader

`DomExtractionTools` is a specialized class that uses [`@mcp-b/smart-dom-reader`](/packages/smart-dom-reader/overview) to extract page content from within the extension context. It injects the smart-dom-reader bundle into the target tab via `chrome.userScripts.execute` and returns structured markdown.

The extraction follows a progressive workflow designed for token efficiency:

1. **`dom_extract_structure`** returns a high-level page outline (headers, navigation, regions).
2. **`dom_extract_region`** extracts interactive elements or full content from a specific CSS selector.
3. **`dom_extract_content`** extracts readable text, headings, lists, and tables from a region.

This progressive approach lets agents start with minimal tokens and drill into areas of interest.

## Use cases

This package is for Chrome extension developers who want to give AI agents access to browser capabilities that web pages cannot provide. Typical use cases include:

- AI assistants that manage tabs, organize bookmarks, or search browsing history
- Agents that read and modify extension storage
- Browser automation extensions that use scripting tools to interact with pages
- Extensions that combine DOM extraction with tab management for page-aware agent workflows

<Note>
  `@mcp-b/extension-tools` runs in the extension background script (service worker) context. If you are building tools for a regular webpage, use [`@mcp-b/global`](/packages/global/overview) instead. For DevTools Protocol access without an extension, see [`@mcp-b/chrome-devtools-mcp`](/packages/chrome-devtools-mcp/overview).
</Note>

## API reference

### BaseApiTools

All API tool classes extend `BaseApiTools<TOptions>`:

```typescript
abstract class BaseApiTools<TOptions> {
  protected abstract apiName: string;

  constructor(server: McpServer, options: TOptions);

  abstract checkAvailability(): ApiAvailability;
  abstract registerTools(): void;

  register(): void;

  protected shouldRegisterTool(toolName: string): boolean;
  protected formatError(error: unknown): CallToolResult;
  protected formatSuccess(message: string, data?: unknown): CallToolResult;
  protected formatJson(data: unknown): CallToolResult;
}
```

| Method | Description |
|--------|-------------|
| `register()` | Checks API availability, then calls `registerTools()` if available |
| `checkAvailability()` | Returns `{ available: boolean, message: string }` |
| `shouldRegisterTool(name)` | Returns `false` if the option for that tool is explicitly `false` |
| `formatError(error)` | Wraps an error into a structured `CallToolResult` with `isError: true` |
| `formatSuccess(message, data?)` | Returns a success `CallToolResult` with optional JSON data |

### Chrome API registry

The `ChromeApi` enum and `CHROME_API_REGISTRY` constant map every Chrome Extension API to its metadata:

```typescript
interface ChromeApiMetadata {
  minChromeVersion?: number;
  platform: 'all' | 'chromeos';
  channel: 'stable' | 'dev';
  manifestVersions: (2 | 3)[];
  requiresPolicy: boolean;
  foregroundOnly: boolean;
  contexts: ('background' | 'content' | 'devtools' | 'all')[];
  namespace: string;
}
```

### Helper functions

| Function | Description |
|----------|-------------|
| `isApiAvailable(api, context, manifestVersion, platform?, chromeVersion?)` | Check if an API is available in the current context |
| `getAvailableApis(context, manifestVersion, platform?, chromeVersion?)` | Get all available APIs for a context |
| `getChromeApiReference(api)` | Get the runtime `chrome.*` object for an API |

### Permission requirements

Declare required permissions in your extension's `manifest.json`. APIs like `i18n`, `runtime`, and `extension` work without explicit permissions. Enterprise APIs require force-install via enterprise policy.

```json "manifest.json (subset)"
{
  "manifest_version": 3,
  "permissions": ["tabs", "bookmarks", "storage", "scripting"],
  "host_permissions": ["<all_urls>"]
}
```
