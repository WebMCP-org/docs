`@mcp-b/chrome-devtools-mcp` is an MCP server that connects AI agents to Chrome through the [Chrome DevTools Protocol](https://chromedevtools.github.io/devtools-protocol/) (CDP). It exposes 27 tools for browser automation (navigation, clicking, screenshots, network inspection, performance tracing) and adds WebMCP integration so agents can discover and call tools registered on webpages.

This package is a fork of Google's [ChromeDevTools/chrome-devtools-mcp](https://github.com/ChromeDevTools/chrome-devtools-mcp) with WebMCP support layered on top.

## How it works

The server runs as a Node.js process that communicates with your MCP client over stdio. On startup, it connects to a Chrome instance (or launches one) and opens a CDP session. Each CDP command is wrapped as an MCP tool that agents call by name.

For WebMCP integration, the server monitors open pages for `navigator.modelContext`. When a page loads [`@mcp-b/global`](/packages/global/overview), the server detects registered tools and tracks them through a `WebMCPToolHub`. Agents discover these tools with `list_webmcp_tools` and call them with `call_webmcp_tool`.

```
AI Client  --stdio-->  chrome-devtools-mcp  --CDP-->  Chrome
                                |
                                +-- list_webmcp_tools --> discovers page tools
                                +-- call_webmcp_tool  --> executes page tools
```

## Quick start

Add the server to your MCP client configuration:

<CodeGroup>

```json "Claude Desktop / Cursor / Windsurf"
{
  "mcpServers": {
    "chrome-devtools": {
      "command": "npx",
      "args": ["-y", "@mcp-b/chrome-devtools-mcp@latest"]
    }
  }
}
```

```bash "Claude Code"
claude mcp add chrome-devtools npx @mcp-b/chrome-devtools-mcp@latest
```

</CodeGroup>

The server connects to a running Chrome instance automatically (Chrome 144+). If no running instance is found, it launches a new one. Pass `--browserUrl=http://127.0.0.1:9222` to connect to a specific instance, or `--headless=true` to run without a visible window.

## Key capabilities

The server provides 27 tools organized into seven categories:

| Category | Tools | What agents can do |
|----------|-------|--------------------|
| Input (8) | `click`, `fill`, `fill_form`, `hover`, `drag`, `press_key`, `handle_dialog`, `upload_file` | Interact with page elements |
| Navigation (6) | `navigate_page`, `new_page`, `close_page`, `list_pages`, `select_page`, `wait_for` | Move between pages and tabs |
| Debugging (5) | `evaluate_script`, `take_screenshot`, `take_snapshot`, `list_console_messages`, `get_console_message` | Inspect page state and capture output |
| Performance (3) | `performance_start_trace`, `performance_stop_trace`, `performance_analyze_insight` | Profile runtime performance |
| Network (2) | `list_network_requests`, `get_network_request` | Monitor HTTP traffic |
| Emulation (2) | `emulate`, `resize_page` | Simulate devices and network conditions |
| WebMCP (1) | `list_webmcp_tools` | Discover tools registered on pages |

## WebMCP tool discovery

When a page registers tools through `@mcp-b/global`, this server discovers them automatically. The `list_webmcp_tools` tool returns tool definitions (name, description, input schema) from the current page. Use `call_webmcp_tool` to invoke any discovered tool by name.

This means agents can both automate the browser (click, type, navigate) and call structured tool APIs exposed by the page, all through a single MCP server.

For a working example of this workflow, see the [Chrome DevTools Quickstart](https://github.com/WebMCP-org/chrome-devtools-quickstart).

## When to use this package

- You want AI agents to automate Chrome: navigate pages, fill forms, take screenshots, inspect network traffic.
- You need agents to discover and call [WebMCP tools](/explanation/what-is-webmcp) from live pages through DevTools.
- You are building coding-agent workflows that speak MCP over stdio (Claude Code, Cursor, GitHub Copilot).

If you only need to forward page tools into a desktop MCP client without browser automation, use [`@mcp-b/webmcp-local-relay`](/packages/webmcp-local-relay/overview) instead.

## API reference

### Configuration options

Pass options via the `args` array in your MCP client config.

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `--autoConnect` | boolean | `true` | Automatically connect to a running Chrome instance (Chrome 144+) via `chrome://inspect/#remote-debugging` |
| `--browserUrl`, `-u` | string | | URL of a running debuggable Chrome instance (e.g. `http://127.0.0.1:9222`) |
| `--wsEndpoint`, `-w` | string | | WebSocket endpoint for a running Chrome instance |
| `--wsHeaders` | string | | Custom headers for WebSocket connection in JSON format. Only works with `--wsEndpoint` |
| `--headless` | boolean | `false` | Run Chrome in headless mode |
| `--executablePath`, `-e` | string | | Path to a custom Chrome executable |
| `--isolated` | boolean | `false` | Use a temporary user-data-dir, cleaned up on close |
| `--userDataDir` | string | `$HOME/.cache/chrome-devtools-mcp/chrome-profile` | Path to the Chrome user data directory |
| `--channel` | `stable` \| `canary` \| `beta` \| `dev` | `stable` | Chrome channel to use |
| `--logFile` | string | | Path to write debug logs. Set `DEBUG=*` for verbose output |
| `--viewport` | string | | Initial viewport size (e.g. `1280x720`). Max 3840x2160 in headless mode |
| `--proxyServer` | string | | Proxy server passed as `--proxy-server` to Chrome |
| `--acceptInsecureCerts` | boolean | `false` | Ignore self-signed and expired certificate errors |
| `--chromeArg` | array | | Additional Chrome launch arguments |
| `--ignoreDefaultChromeArg` | array | | Disable specific default Chrome launch arguments |
| `--categoryEmulation` | boolean | `true` | Set to `false` to exclude emulation tools |
| `--categoryPerformance` | boolean | `true` | Set to `false` to exclude performance tools |
| `--categoryNetwork` | boolean | `true` | Set to `false` to exclude network tools |

```json "Example: headless with canary channel"
{
  "mcpServers": {
    "chrome-devtools": {
      "command": "npx",
      "args": [
        "@mcp-b/chrome-devtools-mcp@latest",
        "--channel=canary",
        "--headless=true",
        "--isolated=true"
      ]
    }
  }
}
```

### Tools (27)

#### Input automation (8 tools)

| Tool | Description | Required parameters |
|------|-------------|---------------------|
| `click` | Click on an element | `uid` |
| `drag` | Drag an element onto another | `from_uid`, `to_uid` |
| `fill` | Type text into an input, textarea, or select | `uid`, `value` |
| `fill_form` | Fill multiple form elements at once | `elements` (array) |
| `handle_dialog` | Accept or dismiss a browser dialog | `action` (`accept` or `dismiss`) |
| `hover` | Hover over an element | `uid` |
| `press_key` | Press a key or combination (e.g. `Control+A`) | `key` |
| `upload_file` | Upload a file through a file input | `uid`, `filePath` |

#### Navigation automation (6 tools)

| Tool | Description | Required parameters |
|------|-------------|---------------------|
| `close_page` | Close a page by index | `pageIdx` |
| `list_pages` | List open browser pages | (none) |
| `navigate_page` | Navigate current page (URL, back, forward, reload) | (varies by `type`) |
| `new_page` | Open a new page | `url` |
| `select_page` | Select a page as context for future tool calls | `pageIdx` |
| `wait_for` | Wait for text to appear on the page | `text` |

#### Emulation (2 tools)

| Tool | Description | Required parameters |
|------|-------------|---------------------|
| `emulate` | Emulate CPU throttling, geolocation, or network conditions | (all optional) |
| `resize_page` | Resize the page viewport | `width`, `height` |

#### Performance (3 tools)

| Tool | Description | Required parameters |
|------|-------------|---------------------|
| `performance_start_trace` | Start a performance trace recording | `autoStop`, `reload` |
| `performance_stop_trace` | Stop the active performance trace | (none) |
| `performance_analyze_insight` | Get details on a specific performance insight | `insightSetId`, `insightName` |

#### Network (2 tools)

| Tool | Description | Required parameters |
|------|-------------|---------------------|
| `list_network_requests` | List all network requests since last navigation | (all optional) |
| `get_network_request` | Get a specific network request by ID | (optional `reqid`) |

#### Debugging (5 tools)

| Tool | Description | Required parameters |
|------|-------------|---------------------|
| `evaluate_script` | Execute JavaScript in the current page | `function` |
| `list_console_messages` | List console messages | (all optional) |
| `get_console_message` | Get a console message by ID | `msgid` |
| `take_screenshot` | Take a screenshot (PNG, JPEG, or WebP) | (all optional) |
| `take_snapshot` | Take a text snapshot based on the a11y tree | (all optional) |

#### WebMCP (1 tool)

| Tool | Description | Required parameters |
|------|-------------|---------------------|
| `list_webmcp_tools` | List website MCP tools across all pages (diff-aware) | (all optional) |

When a page has WebMCP tools registered via `@mcp-b/global`, those tools are automatically registered as first-class MCP tools with prefixed names (e.g. `webmcp_localhost_3000_page0_search_products`).

| Prompt | Description | Arguments |
|--------|-------------|-----------|

### Dynamic tool registration

WebMCP tools from webpages are registered as first-class MCP tools. Client support for `tools/list_changed` notifications:

| Client | Dynamic updates |
|--------|----------------|
| Claude Code | Yes |
| GitHub Copilot | Yes |
| Gemini CLI | Yes |
| Cursor | No (use `list_webmcp_tools` to poll) |
| Cline | Partial |

The `list_webmcp_tools` tool is diff-aware: the first call returns the full list, subsequent calls return only changes. Pass `full: true` to force the complete list.

### Connecting to a running Chrome instance

Two approaches:

1. **Automatic (Chrome 144+)**: Enable remote debugging at `chrome://inspect/#remote-debugging`. The server connects automatically with `--autoConnect` (enabled by default).

2. **Manual via port**: Start Chrome with `--remote-debugging-port=9222` and a non-default `--user-data-dir`, then pass `--browserUrl=http://127.0.0.1:9222` to the MCP server.

<Warning>
Enabling the remote debugging port allows any local application to control the browser. Do not browse sensitive websites while the debugging port is open.
</Warning>

### User data directory

Default paths:

| Platform | Path |
|----------|------|
| Linux / macOS | `$HOME/.cache/chrome-devtools-mcp/chrome-profile-$CHANNEL` |
| Windows | `%HOMEPATH%/.cache/chrome-devtools-mcp/chrome-profile-$CHANNEL` |

The directory persists between runs. Set `--isolated=true` to use a temporary directory that is cleaned up when the browser closes.

### Known limitations

- Operating system sandboxes (macOS Seatbelt, Linux containers) can prevent Chrome from launching. Use `--browserUrl` to connect to an externally started Chrome instance.
- The MCP server exposes browser content to MCP clients. Avoid browsing sensitive data during a session.
